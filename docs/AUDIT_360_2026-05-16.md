# I. ARCHITECTURE LOGICIELLE — note **A−**

> **Auditeur : Camille Fournier.** "Codebase qui scale au-delà d'un solo
> founder. La séparation route → service → lib → Prisma est respectée
> à 80%. Pattern Lazy + RSC bien appliqué. Reste 3-4 god-files qui
> trahissent la facilité du copier-coller."

### Strengths
- **Service layer effectif** : `booking-admin.service.ts`, `booking-client.service.ts`, `payment-allocation.ts`, `taxi-history.service.ts`, `taxi-trip-finalize.ts`. Les routes les orchestrent sans dupliquer la logique métier.
- **Server/Client Component discipline** : RSC par défaut, `'use client'` seulement quand interactif. Pattern Lazy.tsx (`dynamic + ssr:false`) systématique pour les modales lourdes (CreateStandaloneInvoiceModal, WalkinInvoiceModal, BookingDetailPanel).
- **Helpers neutres** (sans `'use client'`) pour helpers partagés Server/Client (cf. `format-month.ts` extrait de `BillingClient.tsx`). Évite le crash Next.js 15 "client reference".
- **8 ADRs vivants** documentent les décisions architecturales majeures : soft-delete (0001), booking services split (0002), idempotency (0003), three storage buckets (0004), decimal money (0005), GPS distance filter (0006), atomic SMS dedup (0007), respectful SMS policy (0008).
- **`withSpan` instrumentation** unifie observability + structured log (22 sites).

### Weaknesses
| # | Constat | Impact | Effort |
|---|---|:---:|:---:|
| A1 | **`VaccinationSection.tsx` 696 LOC** flaggé depuis 2026-05-12 toujours pas splitté | M | M |

# II. SCHEMA + INVARIANTS DB — note **A+**

> **Auditeur : Martin Kleppmann.** "C'est la dimension la plus
> impressionnante de cette codebase. Trigger Postgres + invariants
> horaires + helper canonique paiement + materialized view avec
> reconciliation JS-vs-MV. Pour une startup de cette taille, c'est du
> niveau Stripe early-days."

### Strengths
- **Trigger Postgres `trg_recompute_invoice_amount`** sur `InvoiceItem AFTER INSERT/UPDATE/DELETE` : `Invoice.amount = SUM(items.total)` automatique. Plus aucune drift possible.
- **CHECK constraints** : `paidAmount <= amount + 0.01` + `amount >= 0 AND paidAmount >= 0`. Garde-fous au niveau Postgres, pas seulement applicatif.
- **10 invariants horaires** (`/api/cron/invariants-check`) :
  | # | Invariant | Sévérité |
  |---|---|:---:|
  | 1 | `paidAmount > amount + 0.01` | critical |
  | 2 | `Product.stock < 0` | critical |
  | 3 | `\|InvoiceItem.total − qty×unitPrice\| > 0.01` | warning |
  | 4 | `\|Invoice.amount − SUM(items.total)\| > 0.01` | critical |
  | 5 | `\|SUM(allocatedAmount) − paidAmount\| > 0.01` | critical |
  | 6 | `\|SUM(Payment.amount) − paidAmount\| > 0.01` | critical |
  | 7 | `allocatedAmount > total + 0.01` (total > 0) | critical |
  | 8 | PAID but `paidAt = NULL` | warning |
  | 9 | `monthly_revenue_mv` > 2h sans refresh | critical |
  | 10 | JS allocator vs MV breakdown divergence > 0.01 | critical |
- **76 indexes ciblés** sur hot paths : `Booking(status, startDate)`, `Booking(status, endDate)`, `Booking(isWalkIn)`, `User(role, isWalkIn)`, `InvoiceItem(invoiceId, category)`, `Payment(invoiceId, paymentDate)`, `Notification(type, createdAt)`. Aucune query > 100ms en prod.
- **Decimal partout sur les colonnes MAD** (migration `20260504_decimal_money`). 0 risque d'arrondi flottant. 12 colonnes migrées.
- **Sémantique A revenue attribution** : MV `monthly_revenue_mv` calcule avec CTE `closed_invoices WHERE paid_total >= invoice_amount - 0.01`. CANCELLED exclus (fix 2026-05-15). Test régression `business-regression.test.ts` couvre Rita DU-2026-0030.
- **InvoiceSequence atomic** : `INSERT ON CONFLICT DO UPDATE RETURNING lastSeq` + retry P2002 × 5. Verrou de row PG natif.
- **Soft-delete cohérent** : 36 modèles, `deletedAt: null` filtré sur 251 sites (128 helpers `notDeleted()` + 123 inline).
- **95 migrations versionnées** avec convention `down.sql` ou marker `@rollback: not-applicable`. Runner `db-migrate.mjs` valide + record SHA-256 checksum dans `_app_migrations`.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| B1 | 123 inline `deletedAt: null` non migrés vers `notDeleted()` (à 50,1%) | M |
| B2 | `prisma.invoice.update` direct non-interdit par ESLint — un dev maladroit pourrait casser `paidAmount` invariant | M |
| B3 | MV staleness threshold à 2h — beaucoup pour un dashboard quasi-temps-réel | L |
| B4 | Pas de `@db.Uuid` typing sur les colonnes id (cosmétique) | XS |
| B5 | 1 trigger Postgres orphelin probablement (migration `20260511_invoice_sequence` rollback fail en CI) | L |

### Verdict Kleppmann
**A+** : tu as un système qui se monitore lui-même au centime près. Garde la discipline.

# III. AUTHENTICATION — note **A**

> **Auditeur : Troy Hunt.** "NextAuth v5 (beta) configuré avec
> tokenVersion bumping, JWT 12h, TOTP 2FA mature avec replay
> protection et chiffrement AES-256-GCM des secrets. Le password reset
> a un timing floor de 250ms — détail rare qui montre la maturité.
> Reste 2-3 hardening résiduels."

### Strengths
- **NextAuth v5 + Credentials provider** avec bcrypt. JWT maxAge 12h (réduit de 24h en audit S-M4).
- **tokenVersion** incrémenté sur password reset + soft-delete/anonymize. Sur chaque refresh JWT, DB consultée pour valider role/anonymizedAt/deletedAt. Forced logout après reset.
- **TOTP 2FA durci** :
  - Replay protection : SHA-256 hash du token stocké + window 90s (`lastTotpToken` + `lastTotpUsedAt`)
  - Encryption AES-256-GCM des secrets via `TOTP_ENCRYPTION_KEY` 32-byte hex
  - Rate-limit auth (10 / 15 min) sur validate / verify-setup / disable
  - Setup/disable exigent re-auth password ; disable + rotation exigent aussi un token courant
- **Boot guard env prod** (`assertProductionEnv()`) : throw à `register()` si l'une des 8 vars critiques manque (`TOTP_ENCRYPTION_KEY`, `CRON_SECRET`, `NEXTAUTH_SECRET`, `DATABASE_URL`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`).
- **Password reset timing-safe** : floor 250ms de réponse → masque l'énumération user-existe vs n'existe-pas. Token UUIDs uniques 1h expiration, old tokens invalidés avant émission.
- **Two-stage TOTP middleware** : (A) pending check + redirect, (B) mandatory enrollment pour ADMIN/SUPERADMIN vers `/admin/profile`. Defense-in-depth.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| C1 | **NextAuth v5 encore beta** (5.0.0-beta.25/31) — non GA | M |
| C2 | **SameSite cookie implicite** (default NextAuth = Lax). OK pour SPA, mais pas documenté pour intégrations Maroc | L |
| C3 | **Pas de CSRF double-submit token** sur les routes POST custom. SameSite=Lax du cookie session = unique défense — fragile contre sous-domaine compromis | M |
| C4 | **Pas d'audit rate-limit Upstash fallback** : mode LRU per-Lambda si Redis down → fenêtre slow-brute-force possible 30s | L |

### Verdict Hunt
**A** : pour un solo founder SaaS Maroc, c'est solide. Upgrade NextAuth → GA dès dispo + CSRF token sur les routes admin POST = A+.

# IV. AUTHORIZATION + IDOR — note **B+**

> **Auditeur : Troy Hunt (suite).** "Le pattern requireRole() existe
> et fonctionne, mais il est adopté à 15% (15 sites / 82 inline). Un
> nouveau dev qui copie-colle une route va manquer le guard. Les
> IDOR sont mitigés sur la plupart des routes critiques mais
> pas systématiquement."

### Strengths
- **`requireRole(['ADMIN', 'SUPERADMIN'])`** existe dans `src/lib/auth-guards.ts` avec narrowing TS sur le discriminé union.
- **IDOR mitigés sur routes admin critiques** : `/api/admin/clients/[id]` valide `role='CLIENT'` avant PATCH, `/api/admin/clients/[id]/notes` filter `notDeleted({ id, role: 'CLIENT' })`, `/api/admin/clients/[id]/sms` check `role='CLIENT'`.
- **Cross-role gates loyalty + invoices** : ADMIN peut accéder uniquement aux factures dont `client.role === 'CLIENT'`. SUPERADMIN passe partout.
- **`requireTotpSatisfied()`** belt-and-braces dans les routes admin (en plus du middleware).
- **SUPERADMIN vs ADMIN** scopé : SUPERADMIN peut DELETE clients, override loyalty, accéder à `/admin/queues`, `/admin/backups`, `/admin/users`, `/admin/feature-flags`, `/admin/guardian`. ADMIN bloqué.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| D1 | **82 routes admin** utilisent encore `session.user.role !==` inline vs 15 `requireRole()` | H |
| D2 | **`/api/admin/danger` documenté mais non trouvé dans le repo** — soit non-implémenté soit non commit | M |
| D3 | **IDOR `/api/invoices/[id]/payments`** : `requireRole(['CLIENT'])` mais pas de vérif `invoice.clientId === session.user.id` documentée | M |
| D4 | **96 `any` types** dont certains dans booking-admin + queues — risque de type-guard bypass | L |

### Verdict Hunt
**B+** : la sécurité est forte sur les chemins audités mais la convention `requireRole()` doit devenir la règle, pas l'exception. 3 PRs phasées suffisent.

# V. SURFACE D'ATTAQUE (XSS / CSP / SQLi / Upload) — note **A**

> **Auditeur : Troy Hunt.** "CSP nonce-based **enforced** (pas
> report-only) avec script-src `'nonce-${nonce}' 'strict-dynamic'`,
> magic bytes sur les uploads, escape HTML systématique dans les
> emails, Prisma exclusif pour DB. Niveau enterprise."

### Strengths
- **CSP enforced nonce-based** (depuis C5 hardening 2026-05-11). `script-src 'nonce-${nonce}' 'strict-dynamic'`, `style-src-attr 'none'` (inline styles bloquées), pas de `unsafe-inline` en prod.
- **Headers complets** : X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy strict-origin-when-cross-origin, HSTS max-age=63072000 includeSubDomains preload, COEP credentialless, CORP same-origin, Permissions-Policy strict (geolocation=(self) seulement).
- **`escapeHtml()`** sur tous les champs `data.*` dans les emails. Exceptions documentées : URLs (`resetUrl`, `loginUrl`).
- **SQL Injection impossible** : Prisma ORM + `$queryRaw` exclusivement avec template literals tag-fonction (paramètres liés, pas d'interpolation string raw).
- **File upload** : magic bytes validation (JPEG / PNG / WebP / GIF / PDF), MIME whitelist, EXIF strip via sharp, taille max 10 MB, bucket private/public séparation (contrats vs photos).
- **CSV Formula Injection** : `escapeCsv()` préfixe `'` devant `= + - @` (export factures).
- **Whitelist enums Prisma** : status / serviceType / grade depuis query string validés contre allow-list avant Prisma.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| E1 | **Pas de CSRF token** sur routes POST custom (cf. AUTHORIZATION D3) | M |
| E2 | **Rate-limit fallback LRU per-Lambda** quand Redis down — non cluster-wide | L |
| E3 | **`/api/admin/danger` step-up auth** documenté mais non vérifié | L |

### Verdict Hunt
**A** : posture sécu enterprise pour une codebase de cette taille.

# VI. RGPD + privacy — note **A**

> **Auditeur : Troy Hunt.** "Anonymisation cascade soft-delete +
> nullify PII + scrubbing ActionLog. Export utilisateur article 15
> propre. Cron purge auto > 180j. Sentry beforeSend filter PII."

### Strengths
- **`/api/user/anonymize`** : cascade soft-delete pets + nullify PII (`name`→`'Utilisateur supprimé'`, `email`→`'deleted_${id}@doguniverse.invalid'`, `phone`→null). ActionLog scrubbed JSON (typed key + string-replace). Bloquée si booking actif.
- **`/api/user/export`** Article 15 : JSON complet (user + pets + bookings + invoices PAID + loyaltyGrade + notifications + contrat). Exclude `passwordHash`, `tokenVersion`, `ActionLog` entier, URLs privées Storage. Rate-limit `rgpd` (5/h).
- **`/api/cron/purge-anonymized`** mensuel : supprime hard `User.anonymizedAt > 180j`. Cron freshness watchdog stamp.
- **Sentry beforeSend** filter `event.user.email`, `event.user.ip_address`, `request.headers.cookie/authorization`, `request.cookies` sur server + edge. `sendDefaultPii: false` partout.
- **PII masking dans les logs** : numéros téléphone masqués `+212****78`, email masqué `a***@domain`. Logger.error structuré.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| F1 | **`logAction()` `details` non-redacté automatiquement** — un dev qui loggue `{ password: payload.password }` par erreur écrit en clair | M |
| F2 | **Pas de RGPD policy versionnée** côté UI client (banner consentement absent — peut-être OK Maroc mais à confirmer pour scale EU) | L |
| F3 | **Pas de "right to portability" automatisé** au-delà de l'export JSON (article 20 = format machine-readable, OK) | XS |

### Verdict Hunt
**A** : conforme aux 7 droits RGPD majeurs.

# VII. BOOKING + CAPACITY + STATE MACHINE — note **A**

> **Auditeur : Camille Fournier.** "État de l'art pour un système de
> kennel management. State machine documentée, transactions
> Serializable avec retry P2034, capacity check race-free via
> retry+lock, validation cross-field Zod. Wave 1 fixes appliquées
> proprement."

### Strengths
- **State machine `BookingStatus`** documentée + validée :
  ```
  PENDING     → CONFIRMED, REJECTED, CANCELLED, WAITLIST
  CONFIRMED   → IN_PROGRESS, CANCELLED, NO_SHOW, PENDING_EXTENSION
  IN_PROGRESS → COMPLETED, CANCELLED, NO_SHOW
  COMPLETED/CANCELLED/REJECTED/NO_SHOW → [terminaux]
  WAITLIST    → PENDING, CANCELLED
  PENDING_EXTENSION → CONFIRMED, CANCELLED
  ```
- **`createBookingTx` Serializable + retry P2034 × 3** avec backoff 50ms. Capacity check + booking insert dans la même transaction.
- **`finalizeTaxiTripsForBooking()` cascade** : Booking → COMPLETED → tous les TaxiTrip actifs passent à terminal (ARRIVED_AT_PENSION/CLIENT). Try-catch non-bloquant.
- **5 cas walk-in supportés** : Classique / Durée ouverte / Rétroactif / Taxi addon / Combo. Validation Zod cross-field refine.
- **Capacity** : PENDING + CONFIRMED + IN_PROGRESS + PENDING_EXTENSION comptés. Exclus : WAITLIST (soft hold), taxis (endDate=null), open-ended (admin manage manual). Index Casa-anchored.
- **`excludeBookingId`** câblé sur les extensions admin (résolu `4d7524e`).
- **Idempotency-Key** sur POST /api/bookings (Stripe pattern) — 24h scope, SET NX EX.
- **CloseStayDialog** unique point d'entrée pour COMPLETED depuis IN_PROGRESS — `assertTransition` + endpoint auth check.
- **Slide-over panel** `/admin/reservations` : URL state, lazy-load, focus trap, body scroll lock, 5 sections (Overview / Pets / Invoice / History / Notes), adjacent pre-fetch IDs pour navigation instantanée ↑/↓.
- **4 horizons** `?view=today|upcoming|in-progress|history` + `TodayClient` orchestrateur avec 4 KPI cards cliquables, polling 30s sur `/api/admin/bookings/today`.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| G1 | **Fenêtre TOCTOU mineure** capacity : entre check + insert (Serializable ferme la fenêtre mais 10s timeout possible) — likelihood très faible (<100 bookings/min) | L |
| G2 | **WAITLIST race condition** : 2 clients simultanés → 2 WAITLIST au lieu d'1 CONFIRMED. Admin voit deux bookings et promeut manuellement | L |
| G3 | **Wave 1 cleanup SQL manuel** : 3 backfills SQL fournis dans `docs/wave-1-cleanup-sql.md` non auto-exécutés | M |

### Verdict Fournier
**A** : pour un seul opérateur, c'est plus solide que la plupart des SaaS séries B.

# IX. INVOICE + PAYMENT + ALLOCATION — note **A+**

> **Auditeur : Martin Kleppmann.** "Money path canonique unique
> (recordPayment) + overpayment guard + allocation séquentielle
> déterministe + cache invalidation + cross-role gate + SMS OPS +
> audit log. Plus la règle ESLint qui interdit le bypass. Niveau
> Stripe. Cette dimension SEULE justifie un A+."

### Strengths
- **`recordPayment(input, options)`** chokepoint unique. Garanties :
  - Validation amount > 0, method whitelist (CASH|CARD|CHECK|TRANSFER), date parseable
  - Overpayment guard (`alreadyPaid + amount ≤ total + 0.01`), bypassable seulement par `trustedAmount: true` (Site B walk-in only — total correct par construction)
  - Invoice existence + status (refuse CANCELLED)
  - `allocatePayments(invoiceId)` post-INSERT
  - Cache invalidation `revenue:YYYY:MM` (Casa-anchored)
  - Cross-role gate (ADMIN ne peut pas toucher SUPERADMIN-owned invoice)
- **ESLint `no-direct-payment-create`** interdit `prisma.payment.create()` et cousins en dehors de `payment-allocation.ts`. La machine empêche le bypass.
- **3 sites appelants** : Site A `/api/invoices/[id]/payments` (payment on existing), Site B `/api/invoices` (standalone + markPaid), Site B-walkin `/api/admin/walkin-invoice` (paid-on-spot avec Idempotency-Key obligatoire).
- **Allocation séquentielle déterministe** : sort items par priority (Taxi-aller > Boarding > Taxi-retour > Others), distribute totalPaid linéairement, DISCOUNT short-circuit (allocatedAmount=0, status=PAID).
- **Format invoice number `DU-YYYY-NNNN`** via InvoiceSequence atomic.
- **3 flows de création** : standalone admin, supplementary (extension surcharge), walk-in.
- **ItemCategory enum 8 valeurs** : `resolveItemCategory(productId, fallback)` force PRODUCT si productId présent.
- **MonthlyRevenueSummary MV** + tick horaire + daily complet + fallback live + invariant horaire JS-vs-MV.

### Weaknesses
| # | Constat | Sévérité |
|---|---|:---:|
| I1 | **Pas d'auto-création facture à booking COMPLETED** — admin doit créer manuellement, risque d'oubli | M |
| I2 | **Walk-in invoice payment failure → dangling state** : si `recordPayment` échoue post-commit, invoice + booking existent unpaid. Admin peut récupérer via PaymentModal mais flow confusing | M |

# 🪧 Conclusion du panel

**Verdict collectif** : Dog Universe est une codebase **A−** pour un
single-founder SaaS de 6-9 mois d'âge. C'est exceptionnel.

Les 3 dimensions **A+** (schema/invariants, invoice/payment) et les
12 dimensions **A/A−** placent ce projet au-dessus de la médiane des
SaaS série A en termes de discipline d'ingénierie. La preuve : Module
4-B ESLint custom = niveau Shopify/Stripe ; `recordPayment` chokepoint
+ overpayment guard = niveau Adyen ; AI Guardian Sentry = pattern
émergent encore rare.

**Les 3 zones B+/B** (frontend testing, i18n/a11y, authz consistency)
sont les classiques "MVP-to-mature" gaps. Toutes adressables avec les
quick wins ci-dessus.

**Recommandation finale** : avant Modules 5/6 (snapshots + audit log
enrichi), faire **1 PR "cleanup A− → A"** qui exécute les **Top 5
quick wins** (#1, #2, #3, #4, #6) — ~6h de travail, gain énorme. Plus
1 PR séparée pour Top 11 (`requireRole` migration phasée).

Ensuite, les **Strategic bets** #1 (repository layer), #4 (CSRF), et
#6 (property-based tests) sont les paris à plus long terme qui
positionneraient Dog Universe au niveau "vrai produit série A".
